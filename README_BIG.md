# Unbake Transcription API

Суть задачи: на вход приходит вокальная дорожка после htdemucs v4, на выходе нужен текст песни с таймстемпами, желательно по словам, а не по строкам.

Я бы делал это не как «всегда гоняем один и тот же ASR», а как роутинг из двух сценариев.

Первый сценарий — трек удалось уверенно опознать. Тогда не надо заново распознавать весь текст с нуля. Гораздо надежнее взять уже известный текст и сделать forced alignment по вокалу. Для этой задачи это лучший путь: меньше галлюцинаций, ниже стоимость, лучше тайминг.

Второй сценарий — это кавер, перепевка, демка, измененный текст или просто трек не опознался. Тогда уже идем в полноценную транскрипцию через Whisper/WhisperX.

Именно такой роутинг, на мой взгляд, здесь важнее выбора конкретной модели. Самая частая ошибка в подобных решениях — пытаться решать все одной нейросетью одинаково.

## Какой API я бы оставил

Три ручки достаточно.

`POST /api/v1/transcriptions`
Принимает `s3_url` и опционально `language`.

`POST /api/v1/transcriptions/upload`
Принимает файл напрямую. Это удобно не только для локального теста, но и для внутренних пайплайнов, ретраев и ручной проверки спорных кейсов.

`GET /api/v1/transcriptions/{id}`
Возвращает статус и результат.

Ответ я бы держал на уровне слов:
- `word`
- `start`
- `end`
- `confidence`

Отдельно я бы еще добавил в будущем:
- `source: aligned_lyrics | asr`
- `coverage`
- `language_detected`

Это полезно на клиенте: можно по-разному показывать надежные и сомнительные куски.

## Почему решение не сводится к «просто берем WhisperX»

WhisperX здесь нужен, но не как единственная идея.

Если трек известный, самый качественный вариант — не full transcription, а alignment готового текста к аудио. Это дешевле и лучше бьется с главным требованием из ТЗ: не галлюцинировать.

Почему это важно именно здесь:
- у вас не обычный speech-to-text, а вокал после source separation
- после demucs остаются артефакты
- в песнях много растянутых слов, адлибов, бэков, проглоченных окончаний
- чистый ASR на таких данных начинает домысливать

Поэтому базовая идея такая:
1. пробуем опознать трек
2. если уверенность высокая — берем candidate lyrics и выравниваем
3. если уверенность низкая — уходим в ASR fallback
4. если качество совсем плохое — лучше вернуть частичный результат, чем уверенно придуманный мусор

## Почему я бы использовал ShazamKit как часть пайплайна

В ТЗ отдельно упомянут ShazamKit на iOS, и это не случайно. Я бы этим пользовался как cheap routing step.

ShazamKit сам по себе не решает задачу синка текста, но он отлично отвечает на вопрос: это вообще известный трек или нет.

Если это известный трек, дальше можно:
- получить кандидат на song identity
- подтянуть текст
- запустить forced alignment

Это дает сразу три плюса:
- меньше ошибок в тексте
- меньше нагрузка на GPU
- ниже latency

Если это кавер с измененным текстом, то fingerprint либо не сойдется, либо даст плохую уверенность, и тогда система сама переключится в ASR.

То есть ShazamKit здесь не «еще одна фича», а нормальный способ не тратить дорогую транскрипцию там, где можно обойтись alignment.

## Основной пайплайн

### Вариант A: трек опознан
1. Клиент отправляет вокал
2. Быстрый fingerprint / song ID
3. Находим candidate lyrics
4. Нормализуем текст
5. Делаем forced alignment по словам
6. Возвращаем результат

Это основной и самый выгодный путь для популярных треков.

### Вариант B: трек не опознан или текст не совпадает
1. Клиент отправляет вокал
2. Идем в WhisperX
3. Получаем rough transcription
4. Делаем word-level alignment
5. Отдаем результат с confidence

Это fallback для каверов, демок и кастомных версий.

## Почему не облачные speech APIs

Я бы не брал Google STT / AWS Transcribe / Azure как основной вариант.

Причины простые:
- они хуже чувствуют себя на вокале, чем на обычной речи
- стоимость быстро упирается в лимит из ТЗ
- данные уходят наружу
- качество на нескольких языках и на зашумленном вокале менее предсказуемо

Плюс в этом кейсе важен не просто текст, а именно синхронизация. А это уже задача не уровня «отправил аудио в generic STT и получил ответ».

## Почему не on-device

Потому что тут и по качеству, и по latency, и по энергопотреблению это плохой компромисс. В ТЗ правильно сказано, что туда лучше не идти.

## Alternatives considered

### 1. Whisper large-v2 + WhisperX
Плюсы:
- лучшая open-source база по качеству
- хорошая поддержка multilingual
- нормальный word-level sync через alignment

Минусы:
- для комфортной работы нужен GPU
- full transcription на каждом запросе — не самый дешевый путь

Вывод: нужен как fallback и как основной путь для unknown/covers.

### 2. faster-whisper
Плюсы:
- заметно дешевле по CPU
- быстрый и удобный для продакшна

Минусы:
- если задача упирается в макс��мум качества, large-v2 все равно надежнее
- проблему галлюцинаций сам по себе не решает

Вывод: хороший practical вариант, если цель — быстро и дешево стартовать.

### 3. Только alignment по готовым lyrics
Плюсы:
- очень точно по таймингам
- дешево
- почти нет галлюцинаций

Минусы:
- не работает на каверах с измененным текстом
- не работает на неизвестных треках

Вывод: отличный path, но не единственный. Нужен fallback.

### 4. Полностью облачный провайдер
Плюсы:
- почти не надо думать про инфру

Минусы:
- дорого
- меньше контроля
- хуже переносимость на музыкальный домен

Вывод: для этого кейса не лучший выбор.

## Что я считаю правильным компромиссом

Если расставить приоритеты так, как они заданы в ТЗ, то порядок решений для меня такой:

1. Сначала убираем галлюцинации
2. Потом добиваемся хорошего word sync
3. Потом уже оптимизируем cost и latency

Именно поэтому я не стал бы продавать идею «одна модель решит все». Здесь лучше выглядит гибрид:
- known song -> align known lyrics
- unknown song / rewritten cover -> ASR fallback

Это и качественнее, и дешевле, и честнее по отношению к реальным данным.

## Стоимость за запрос

Если без роутинга всегда гонять full transcription, то cost будет хуже, чем нужно.

Если сделать роутинг, стоимость сильно падает:
- для опознанных треков платим только за alignment
- для сложных кейсов платим за full ASR

На старте я бы считал так:

### Если держать GPU
A10G / 4090 достаточно для старта.
Трек около 3 минут на `large-v2` укладывается примерно в десятки секунд, а не в минуты.
При 100 запросах в день такой сервер укладывается в бюджет.

Грубо по деньгам:
- GPU машина: порядка $0.4–0.6/час в зависимости от провайдера
- при небольшом дневном объеме и без постоянной 100% загрузки бюджет около $50–90/месяц реалистичен

### Если работать только на CPU
Можно, но для продукта это уже заметно хуже по latency. Для тестового задания CPU ок, для реального мобильного сценария я бы не оставлял это как основной режим.

### Оценка в разрезе одного запроса
Если включить routing и не прогонять весь трафик через full ASR, то выйти в диапазон сильно ниже $0.05 за 3 минуты вполне реально. Именно за счет того, что известные треки не требуют полного распознавания заново.

## На каком железе я бы это крутил

Для MVP:
- 1 GPU-воркер на A10G или RTX 4090
- обычный API-инстанс отдельно
- простая очередь задач
- локальный объектный кэш для моделей и alignment assets

Почему так:
- API и воркер лучше разделить, чтобы тяжелая инференс-нагрузка не мешала HTTP-слою
- модель должна грузиться один раз при старте воркера, а не на каждый запрос
- при таком объеме не нужен сложный зоопарк инфраструктуры

Почему не AWS/GCP на старте:
- переплата за тот же GPU
- для small-stage продукта это не дает реальной пользы

Если объем вырастет до 10k запросов в день, тогда уже просто масштабируются воркеры. Архитектуру ради этого заранее усложнять не нужно.

## Что может пойти не так

### 1. Артефакты после htdemucs v4
Это главный источник проблем. Модель слышит не чистый голос, а голос с остатками музыки, ревербом, грязью и иногда фазовыми артефактами.

Что делаю из-за этого:
- предпочитаю alignment, когда это возможно
- не доверяю слепо low-confidence словам
- лучше режу сомнительные хвосты, чем домысливаю текст

### 2. Back vocals и двойной вокал
Формально out of scope, но в реальности сильно влияет на sync. Особенно там, где бэк вступает с другим текстом или с задержкой.

Что делаю:
- учитываю coverage/confidence
- не обещаю идеальный word sync в местах плотного наложения

### 3. Каверы с измененным текстом
Это как раз кейс, на котором ломается наивная идея «всегда берем lyrics из внешнего источника».

Что делаю:
- known lyrics path только при высокой уверенности в match
- иначе сразу fallback в ASR

### 4. Языки
EN обычно самый простой. Хуже будут RU, JP, PL, особенно если вокал грязный и текст поется неразборчиво.

Что делаю:
- считаю качество отдельно по языкам
- не усредняю все в одну красивую цифру

### 5. Latency первого запроса
Первый запрос дольше из-за cold start модели и докачки ассетов.

Что делаю:
- прогреваю воркер
- храню модели локально

## Как я бы оценивал корректность автоматически

Обычный WER здесь нужен, но его мало.

Если считать только WER, можно получить текст более-менее похожий на оригинал, но с плохим таймингом. Для этого продукта это недостаточно.

Я бы считал минимум четыре метрики:

1. `WER` — насколько правильно распознан текст
2. `CER` — полезно для языков и коротких слов
3. `timestamp error` по словам — например median absolute error по start time
4. `precision@200ms` / `precision@300ms` — какой процент слов попал в нужный тайминг с разумным допуском

Плюс еще одна практическая метрика:
5. `coverage` — какой процент слов система вообще вернула с приемлемой уверенностью

Это важно, потому что система, которая честно вернула 82% слов с хорошим таймингом и не выдумала остальные 18%, для продукта лучше, чем система, которая заполнила все 100% мусором.

## Как я бы собирал тестовый датасет

Я бы разбил датасет не только по языкам, но и по типу сложности:
- чистый lead vocal
- вокал с выраженными артефактами
- плотные backing vocals
- известный оригинал
- кавер с тем же текстом
- кавер с измененным текстом

И оценивал бы не одной общей цифрой, а отдельно по корзинам. Иначе легко обмануть себя красивым средним.

Часть датасета я бы обязательно разметил вручную по словам на коротких фрагментах. Да, это долго. Но без этого невозможно честно проверить именно sync, а не просто transcription quality.

## Что уже сделано в этом репозитории

В репозитории есть рабочий API с двумя путями входа:
- по `s3_url`
- по прямой загрузке файла

Есть реальный тест на файле из датасета и полученный word-level output.

Для локальной проверки добавлен sample-трек и Docker-запуск.

## Почему я считаю это решение адекватным именно под это ТЗ

Потому что оно не пытается решить музыкальную задачу как обычный speech-to-text.

Главная идея здесь не в том, чтобы найти «самую умную модель». Главная идея — правильно разделить кейсы:
- где можно надежно выровнять уже известный текст
- где надо честно распознавать с нуля
- где лучше вернуть частичный результат, чем уверенную фантазию

На мой взгляд, именно это и дает здесь нормальный баланс между accuracy, cost и latency.

## Как запустить

```bash
cp .env.example .env
docker build -t unbake-api .
docker run -p 8000:8000 --env-file .env unbake-api
```

Если есть GPU:

```bash
docker run --gpus all -p 8000:8000 --env-file .env unbake-api
```

Для CPU лучше ставить модель полегче. Для GPU — `large-v2`.
