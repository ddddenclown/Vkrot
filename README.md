# Unbake Transcription API

Сделал небольшой прототип API для задачи с вокалом после htdemucs v4. На вход можно дать либо `s3_url`, либо файл напрямую, на выходе получаем текст с таймстемпами по словам.

Что есть сейчас:
- `POST /api/v1/transcriptions`
- `POST /api/v1/transcriptions/upload`
- `GET /api/v1/transcriptions/{id}`
- Basic Auth
- Docker
- проверка на реальном `.m4a` из датасета

Главная мысль у меня была такая: не хочется слепо прогонять всё через одну модель, потому что на песнях с артефактами после demucs она начинает фантазировать.

Поэтому нормальнее выглядит такой путь:
- если трек уверенно опознается, лучше брать готовый текст и делать forced alignment
- если это кавер, перепевка или просто не получилось опознать трек, тогда уже идти в WhisperX

То есть я бы здесь скорее делал routing, а не ставил всё на один full ASR pipeline.

Сам прототип собран на FastAPI. Подробную версию с аргументацией по архитектуре, стоимости, железу, метрикам и рискам вынес в `README_BIG.md`.

Запуск локально:

```bash
cp .env.example .env
docker build -t unbake-api .
docker run -p 8000:8000 --env-file .env unbake-api
```

Если запускать на GPU, лучше ставить `large-v2`. Для CPU разумнее использовать модель полегче.
В папке results/ валяются 9 распознанных треков, для виспер смол очень адекватные результаты
